# Architecture v1 — Astram Traffic Impact System

> **Last Updated**: 2026-06-18
> **Status**: Approved design, pre-implementation

---

## System Overview

```
Incident Reported
       ↓
Feature Pipeline (feature_pipeline.py)
       ↓
Impact Score Prediction (Module 1 — ML Model)
       ↓
Risk Class (Critical / High / Medium / Low)
       ↓
Resource Recommendation (Module 3 — Lookup + Scaling)
       ↓
Diversion Routing (Module 4 — Corridor-based)
       ↓
Dashboard Display (Module 5 — Command Center)
```

---

## Module Breakdown

### Module 1: Event Risk Prediction

**Input**: Raw incident features (cause, location, time, description)
**Output**: Impact score (0–100) → binned into 4 risk classes

**Architecture Choice**: Composite impact score as continuous target, not rule-based classification.

**Why**: Rules map cause→class (lookup table). ML model predicts score from contextual features (time × location × description × cause), learning non-obvious interactions that add value beyond static rules.

**Training Labels**: Generated by `compute_impact_score()` from outcome signals (see `impact_score_definition.md`)
**Model Features**: Input-only signals (no outcome data)

**Candidate Models**:
- LightGBM / XGBoost (tabular features)
- Optional: stacking with MiniLM description embeddings

### Module 2: Congestion Impact Score

**Input**: Predicted impact score from Module 1
**Output**: 0–100 score displayed on dashboard

The score IS the Module 1 output. No separate module needed — the impact score serves double duty as both the risk classifier input and the congestion impact display.

### Module 3: Resource Recommendation

**Input**: `event_cause` + `veh_type` + predicted risk class
**Output**: Resource allocation vector (tow trucks, police, barricades, crew, diversion)

**Architecture**: Lookup table (see `resource_mapping.md`) scaled by:
- Corridor tier multiplier (1.0–1.5×)
- Rush hour multiplier (1.3× during peak)
- Risk class escalation factor

### Module 4: Diversion Routing

**Input**: Corridor + closure status
**Output**: Alternative routes

**Architecture**: Pre-computed diversion templates for Tier 1 corridors. Real-time selection based on incident location and time.

### Module 5: Command Center Dashboard

**Input**: All module outputs
**Output**: Real-time map + alerts + resource status

## Feature Pipeline Architecture

```
Raw CSV
  → Clean (NULL replacement, type casting, coordinate validation)
  → Temporal features (hour, weekday, month, is_weekend, is_night, cyclical encoding)
  → Location features (corridor_tier, geo_cluster, is_corridor)
  → Text features (description → multilingual MiniLM → 384d → PCA → 32d)
  → Historical features (junction_event_count, station_event_count)
  → Target label (compute_impact_score → impact_class)
  → Output: model_ready.parquet
```

## Key Design Decisions

| Decision | Rationale | Doc |
|----------|-----------|-----|
| Composite score, not rule-based classes | AI adds value by learning context, not replicating lookups | `impact_score_definition.md` |
| requires_road_closure as input feature | Leakage audit confirmed it's set at creation time | `validation_findings.md` |
| Multilingual MiniLM for descriptions | 12.2% Kannada, 43.4% location clues, 27.4% severity clues | `validation_findings.md` |
| DBSCAN geo_cluster + police_station | Spatial clusters + inner-city granularity | `validation_findings.md` |
| 3-tier corridor system | Event volume + strategic importance | `corridor_tiering.md` |
| Resource lookup + scaling | Cause-based baseline scaled by context | `resource_mapping.md` |

```
Inference:
  new_incident → feature_pipeline.py → model.predict() → impact_score → risk_class
                                                        → resource_recommendation
                                                        → dashboard_update
```
